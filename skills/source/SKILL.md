# Deep Study — 源码实现

## 概述

Deep Study Source 对源码实现进行系统性深度分析，产出一套结构固定、可自动校验的研究目录，作为长期可查阅的参考资料。

每次研究都会生成 11 个文件，覆盖数据结构、核心函数、内部机制、控制流、端到端追踪和边界条件。固定结构确保不会遗漏关键维度，也让后续阅读者能立即定位到需要的分析。

适用场景：用户明确要求对源码实现做完整深度分析时触发。不适用于快速问答或摘要任务。

## 适用范围

当用户明确要求对以下对象做完整深度分析时使用：

- 开源项目源码
- 框架或库的内部实现
- 系统级软件的核心模块

主题材料应已存在于当前工作区。

## 输出位置

完整分析结果写入：

```text
study/<topic>/
```

当前对话中只需要：

- 在关键阶段同步进度
- 汇报关键发现和未决问题
- 在结束时给出生成文件导航和高层总结

除非用户明确要求，否则不要把整篇分析正文完整打印到对话里。

## 核心关注点

### 1. 核心抽象与不变量

优先找出真正承载系统状态的结构：

- 核心 struct / class / interface / module 分别持有什么状态
- 哪些条件必须始终成立
- 这些不变量是在哪里被建立、检查、破坏、恢复的
- 哪些字段之间存在约束关系，而不是孤立字段说明

如果代码里有 `assert`、边界检查、初始化顺序、特殊 sentinel 值，它们通常比注释更能揭示不变量。

### 2. 关键执行路径

选取最能代表系统本质的路径做追踪：

- 请求、消息、任务或对象从入口到出口如何流动
- 哪些地方做了关键分支决策
- 状态在每一步如何变化
- 哪些路径是 hot path，哪些是异常路径

不要只画 happy path。异常路径、回滚路径、重试路径往往更能揭示设计。

### 3. 并发、同步与一致性

如果项目涉及并发，需要明确：

- 并发单元是什么
- 哪些状态被共享
- 用了什么同步原语
- 锁顺序、原子操作、队列、回调或事件循环如何共同维护一致性
- 出现竞争、乱序或重入时系统如何保持正确

### 4. 错误处理与资源生命周期

重点追踪：

- 错误如何表示、传播、包装、记录
- 哪些错误可恢复，哪些直接终止当前路径
- 内存、句柄、连接、缓存、引用计数、对象所有权如何转移和释放
- 是否存在泄漏、悬挂引用、重复释放、部分初始化失败等风险点

### 5. 扩展边界与配置系统

如果项目支持插件、配置、抽象接口或后端切换，需要回答：

- 哪些接口是稳定边界
- 哪些只是内部实现细节
- 配置如何加载、校验、生效
- 扩展点是否真的隔离了变化，还是把复杂性泄漏给调用方

## 先读什么

开始任何输出前，必须先做两件事：

1. 扫描工作区材料
2. 读取 `references/output-contract.md`

`references/output-contract.md` 定义了固定文件树、固定二级标题、最小条目数、Mermaid 规则、README 规则和完成前必须通过的校验命令。

## 固定输出合同

研究输出目录必须**严格**包含以下文件：

- `00-scope-and-plan.md`
- `01-landscape.md`
- `02-architecture.md`
- `03-core-data-structures.md`
- `04-core-functions.md`
- `05-mechanisms.md`
- `06-control-flow-and-state.md`
- `07-end-to-end-trace.md`
- `08-failure-and-edge-cases.md`
- `09-synthesis.md`
- `README.md`

规则：

- 不允许用其他文件名替代这些文件。
- 不允许把 `03-core-data-structures.md` 和 `04-core-functions.md` 折叠进 `05-mechanisms.md`。
- 不允许跳过某个文件并声称"内容已在别处覆盖"。
- 除 `README.md` 之外，每个阶段文件都必须至少包含一个 Mermaid 图。

## 每个文件必须做什么

- `00-scope-and-plan.md`
  - 固定研究范围、主题分类、核心问题、材料清单和输出计划
- `01-landscape.md`
  - 建立全局地图，明确材料结构、入口点和初始问题
- `02-architecture.md`
  - 解释核心抽象、组件关系、数据流、边界和接口契约
- `03-core-data-structures.md`
  - 显式分析核心数据结构：结构体、类、状态对象，不得只在机制文档中零散提到
- `04-core-functions.md`
  - 显式分析关键函数：调度函数、状态转换函数、关键路径函数，不得只在执行路径中顺带提到
- `05-mechanisms.md`
  - 分析关键内部机制，以及机制之间如何协作
- `06-control-flow-and-state.md`
  - 把跨模块控制流、状态推进和同步点单独讲清楚
- `07-end-to-end-trace.md`
  - 选代表性场景，把系统完整跑一圈
- `08-failure-and-edge-cases.md`
  - 单独分析异常路径、边界条件、清理与资源释放
- `09-synthesis.md`
  - 提炼设计哲学、trade-off、模式和核心结论
- `README.md`
  - 提供所有文件的相对链接、阅读顺序和一句话摘要

## 工作流

### 1. 建立研究目录

创建 `study/<topic>/`，并按固定输出合同规划文件。

### 2. 先做结构化研究，再写文档

不要一边看一边把内容随意散落到各文件里。先明确：

- 这个项目的核心数据结构有哪些
- 关键函数有哪些
- 关键内部机制有哪些
- 最值得追踪的控制流和异常路径是什么

然后把它们分别放进对应文件。

### 3. 显式填充 `03` 和 `04`

这是本 skill 的硬要求。

- `03-core-data-structures.md` 必须显式列出并分析核心结构体、类、状态对象
- `04-core-functions.md` 必须显式列出并分析关键函数

不要把这两类内容吞进 `05-mechanisms.md`。

### 4. 机制、控制流、端到端和边界条件分开写

以下四类内容必须分开：

- 结构（03）
- 函数（04）
- 机制（05）
- 端到端 trace（07）

`08-failure-and-edge-cases.md` 也必须单独存在，不能把异常路径散落在其他章节里就算完成。

### 5. 完成前强制校验

在宣称研究完成前，必须运行：

```bash
python3 deep-study/scripts/validate-study.py study/<topic>
```

如果校验失败：

- 不得宣称完成
- 不得说"主要内容已经有了"
- 必须修正文档并重新运行校验，直到退出码为 `0`

## 优先收集的证据

- 入口函数、核心导出、调度循环
- 核心数据结构定义及其辅助函数
- 测试代码，尤其是 edge case 测试
- `TODO`、`FIXME`、`HACK`、设计注释
- 配置文件、构建脚本、基准测试、性能相关代码

如果 README、注释和真实代码行为不一致，以代码为准，并明确标出差异。

## 深挖时优先问的问题

- 这个状态是谁创建的，谁修改它，谁最终销毁它
- 为什么这里是这个分支条件，而不是另一个看似接近的条件
- 如果失败、超时、竞争、重入、乱序发生，会走哪条路径
- 一个关键值从哪里来，到哪里去，中间被谁改写
- 测试作者最担心的边界情况是什么

## 引用规则

源码分析优先使用 `path:line` 格式引用。

如果涉及 README、设计文档等辅助材料，可用 "文档标题 + 小节标题" 格式。优先使用稳定锚点。

## 不合格示例

以下情况都视为不合格：

- 只写 `mechanisms/` 风格文档，没有 `03-core-data-structures.md`
- 只在执行路径里顺带提到函数，没有 `04-core-functions.md`
- 只列结构体字段名，没有写字段约束、不变量、生命周期关系
- 只给调用链，没有解释状态如何变化
- 只讲 happy path，不讲异常路径、清理和边界
- README 没有完整链接所有文件
- 没跑 validator 就宣称研究完成

## 常见失真

- 只写"模块 A 负责 X，模块 B 负责 Y"，没有解释它们如何协作
- 只列字段名和类型，没有写字段之间的关系和不变量
- 只讲正常流程，不讲异常、清理、恢复和回退
- 只信注释，不核对真实控制流和状态变化
- 把调用链列出来，却没有解释每一步的数据变化

## 完成标准

只有同时满足以下条件，分析才算完成：

1. 固定文件树完整存在
2. 固定二级标题完整存在
3. 最小条目数达到要求
4. 每个阶段文件都有 Mermaid 图
5. 引用符合材料类型
6. `python3 deep-study/scripts/validate-study.py study/<topic>` 退出码为 `0`

如果以上任一条件不满足，禁止宣称完成。
