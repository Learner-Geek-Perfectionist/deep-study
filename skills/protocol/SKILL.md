# Deep Study — 协议规范

## 概述

Deep Study Protocol 对协议规范、线协议实现或消息格式进行系统性深度分析，产出一套结构固定、可自动校验的研究目录，作为长期可查阅的参考资料。

每次研究都会生成 11 个文件，覆盖消息格式、状态机、交互流程、可靠性机制、端到端追踪和边界条件。固定结构确保不会遗漏关键维度，也让后续阅读者能立即定位到需要的分析。

适用场景：用户明确要求对协议规范做完整深度分析时触发。不适用于快速问答或摘要任务。

## 适用范围

当用户明确要求对以下对象做完整深度分析时使用：

- 网络协议规范（RFC、标准文档）
- 线协议实现（编解码、帧处理）
- 消息格式与序列化方案
- 以上对象的规范与实现组合

主题材料应已存在于当前工作区。

## 输出位置

完整分析结果写入：

```text
study/<topic>/
```

当前对话中只需要：

- 在关键阶段同步进度
- 汇报关键发现和未决问题
- 在结束时给出生成文件导航和高层总结

除非用户明确要求，否则不要把整篇分析正文完整打印到对话里。

## 核心关注点

### 1. 消息格式与编码规则

优先解释协议"长什么样"：

- 帧、包、消息、子消息的层次结构
- 每个关键字段的语义，而不只是字段表
- 字节序、对齐、变长编码、TLV、位标志等编码规则
- 扩展字段、可选字段、保留位如何影响兼容性

如果可能，拿一个具体消息实例逐字段拆解，而不是只给抽象表格。

### 2. 状态机

很多协议的本质是状态转换：

- 有哪些状态
- 每个转换由什么事件触发
- 转换时会带来哪些副作用
- 哪些状态是暂态，哪些状态是稳定态
- 哪些转换不可逆，哪些需要对端确认

如果规范没有直接画状态图，需要从多个章节和实现代码中自行拼出来。

### 3. 交互流程

至少搞清楚最具代表性的交互序列：

- 建连、协商、握手、发现、订阅、关闭
- 请求与响应如何关联
- 乱序、重传、确认、超时如何改变流程
- 双方分别在什么时刻更新各自状态

一张好的时序图通常比大段顺序描述更有价值。

### 4. 错误处理、可靠性与恢复

重点追踪：

- 错误码或拒绝原因如何分类
- 超时、重传、确认、去重、幂等如何设计
- 半开连接、丢包、重复包、版本不匹配时会发生什么
- 实现为了现实网络环境做了哪些偏离规范的处理

### 5. 安全与兼容性

如果协议涉及安全或版本演进，必须说明：

- 认证、完整性、加密、防重放如何实现
- 版本协商和能力协商如何进行
- 新旧版本互操作时，哪些字段或状态最容易出错
- 规范有哪些隐含假设，在现实部署中何时会失效

## 先读什么

开始任何输出前，必须先做两件事：

1. 扫描工作区材料
2. 读取 `references/output-contract.md`

`references/output-contract.md` 定义了固定文件树、固定二级标题、最小条目数、Mermaid 规则、README 规则和完成前必须通过的校验命令。

## 固定输出合同

研究输出目录必须**严格**包含以下文件：

- `00-scope-and-plan.md`
- `01-landscape.md`
- `02-architecture.md`
- `03-message-formats.md`
- `04-state-machines.md`
- `05-exchange-sequences.md`
- `06-reliability-and-timing.md`
- `07-end-to-end-trace.md`
- `08-failure-and-edge-cases.md`
- `09-synthesis.md`
- `README.md`

规则：

- 不允许用其他文件名替代这些文件。
- 不允许把 `03-message-formats.md` 和 `04-state-machines.md` 折叠进 `05-exchange-sequences.md`。
- 不允许跳过某个文件并声称"内容已在别处覆盖"。
- 除 `README.md` 之外，每个阶段文件都必须至少包含一个 Mermaid 图。

## 每个文件必须做什么

- `00-scope-and-plan.md`
  - 固定研究范围、主题分类、核心问题、材料清单和输出计划
- `01-landscape.md`
  - 建立全局地图，明确规范文档结构、关键章节和初始问题
- `02-architecture.md`
  - 解释协议分层、组件关系、数据流、边界和接口契约
- `03-message-formats.md`
  - 显式分析消息格式：帧结构、字段语义、编码规则，不得只在交互流程中零散提到
- `04-state-machines.md`
  - 显式分析协议状态机：状态定义、转换条件、副作用，不得只在时序图中顺带提到
- `05-exchange-sequences.md`
  - 分析关键交互流程：握手、协商、传输、关闭，以及流程之间如何衔接
- `06-reliability-and-timing.md`
  - 把超时、重传、确认、去重、流控和拥塞控制单独讲清楚
- `07-end-to-end-trace.md`
  - 选代表性交互场景，把协议完整跑一圈
- `08-failure-and-edge-cases.md`
  - 单独分析异常路径、边界条件、半开连接、版本不匹配和恢复机制
- `09-synthesis.md`
  - 提炼设计哲学、trade-off、模式和核心结论
- `README.md`
  - 提供所有文件的相对链接、阅读顺序和一句话摘要

## 工作流

### 1. 建立研究目录

创建 `study/<topic>/`，并按固定输出合同规划文件。

### 2. 先做结构化研究，再写文档

不要一边看一边把内容随意散落到各文件里。先明确：

- 这个协议的核心消息格式有哪些
- 状态机有哪些状态和关键转换
- 关键交互流程有哪些
- 最值得追踪的异常路径和恢复机制是什么

然后把它们分别放进对应文件。

### 3. 显式填充 `03` 和 `04`

这是本 skill 的硬要求。

- `03-message-formats.md` 必须显式列出并分析每种关键消息的格式和字段语义
- `04-state-machines.md` 必须显式列出并分析协议状态机的每个状态和转换

不要把这两类内容吞进 `05-exchange-sequences.md`。

### 4. 格式、状态机、交互流程和边界条件分开写

以下四类内容必须分开：

- 消息格式（03）
- 状态机（04）
- 交互流程（05）
- 端到端 trace（07）

`08-failure-and-edge-cases.md` 也必须单独存在，不能把异常路径散落在其他章节里就算完成。

### 5. 完成前强制校验

在宣称研究完成前，必须运行：

```bash
python3 deep-study/scripts/validate-study.py study/<topic>
```

如果校验失败：

- 不得宣称完成
- 不得说"主要内容已经有了"
- 必须修正文档并重新运行校验，直到退出码为 `0`

## 优先收集的证据

- 规范中的消息格式定义、状态转换描述、时序要求
- 实现代码中的编码/解码、状态更新、超时与重传逻辑
- 示例报文、测试数据、抓包、互操作测试
- 规范与实现不一致的地方

## 深挖时优先问的问题

- 某个字段为什么存在，如果省略会破坏什么
- 某次状态转换为什么需要这个前置条件
- 如果消息丢失、重复、乱序、延迟，会怎样恢复
- 规范要求和实现选择之间有哪些差距，差距背后的工程理由是什么
- 协议假设的网络和对端行为是什么，这些假设在哪些场景下失效

## 引用规则

协议分析优先使用"文档名 + section / 章节号"格式引用。规范材料通常比行号更适合按章节定位。

涉及实现代码时可用 `path:line` 格式。优先使用稳定锚点，不要把所有材料机械压成同一种引用格式。

## 不合格示例

以下情况都视为不合格：

- 只写交互流程文档，没有 `03-message-formats.md`
- 只在时序图里顺带提到状态，没有 `04-state-machines.md`
- 只列消息字段名，没有写字段语义、约束和编码规则
- 只给交互序列，没有解释状态如何变化
- 只讲 happy path，不讲超时、重传、拒绝和恢复
- README 没有完整链接所有文件
- 没跑 validator 就宣称研究完成

## 常见失真

- 只列消息字段，不解释字段之间的约束和协作方式
- 只讲 happy path，不讲超时、重传、拒绝、恢复
- 把规范文字复述一遍，却没有用实现或实例验证理解
- 忽略兼容性和版本演进，只看当前单一版本
- 只看单个消息，不看整段交互的节奏和状态变化

## 完成标准

只有同时满足以下条件，分析才算完成：

1. 固定文件树完整存在
2. 固定二级标题完整存在
3. 最小条目数达到要求
4. 每个阶段文件都有 Mermaid 图
5. 引用符合材料类型
6. `python3 deep-study/scripts/validate-study.py study/<topic>` 退出码为 `0`

如果以上任一条件不满足，禁止宣称完成。
