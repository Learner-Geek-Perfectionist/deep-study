# Deep Study — 编程语言机制

## 概述

Deep Study Language 对编程语言机制进行系统性深度分析，产出一套结构固定、可自动校验的研究目录，作为长期可查阅的参考资料。

每次研究都会生成 11 个文件，覆盖类型与运行时表示、编译/解释管线阶段、求值规则、运行时机制、端到端追踪和边界条件。固定结构确保不会遗漏关键维度，也让后续阅读者能立即定位到需要的分析。

适用场景：用户明确要求对编程语言机制做完整深度分析时触发。不适用于快速问答或摘要任务。

## 适用范围

当用户明确要求对以下对象做完整深度分析时使用：

- 类型系统及其实现
- 编译器或解释器内部机制
- 运行时系统（GC、调度、FFI）
- 语言语义与求值模型
- 内存模型与并发模型

主题材料应已存在于当前工作区。

## 输出位置

完整分析结果写入：

```text
study/<topic>/
```

当前对话中只需要：

- 在关键阶段同步进度
- 汇报关键发现和未决问题
- 在结束时给出生成文件导航和高层总结

除非用户明确要求，否则不要把整篇分析正文完整打印到对话里。

## 核心关注点

### 1. 语义模型

先回答"这个机制到底是什么意思"：

- 求值规则是什么
- 作用域、绑定、借用、生命周期、类型检查或约束求解如何工作
- 规则是静态生效、运行时生效，还是两者结合
- 哪些边界案例最能暴露规则的真实含义

不要只给抽象定义，必须配具体程序片段或反例。

### 2. 编译或解释管线

从输入到执行，至少说明关键阶段：

- 词法、语法、语义分析分别负责什么
- AST、HIR、MIR、字节码、IR 等中间表示各自承载什么信息
- 哪些规则在哪个阶段落实
- 某个高层语言特性是如何一步步被降解、转换或消除的

如果是解释型语言，也要说明运行前准备阶段和执行期机制如何衔接。

### 3. 运行时表示与执行机制

重点追踪：

- 对象、值、类型信息在运行时如何表示
- 栈、堆、闭包、环境、vtable、字典传递、写屏障、GC 或所有权系统如何运作
- 异常、panic、协程、调度器、FFI 边界如何实现
- 某个语言承诺在运行时靠什么兑现

### 4. 类型系统与约束求解

如果主题涉及类型机制，需要说明：

- 类型信息何时产生、传播、收窄、擦除或单态化
- 约束如何被收集与求解
- 推导失败时编译器如何定位和报告错误
- 泛型、trait、typeclass、protocol、子类型、多态各自靠什么实现

### 5. 对比与 trade-off

语言机制通常是设计取舍的结果：

- 解决了什么问题
- 相比替代方案牺牲了什么
- 它依赖哪些前提条件
- 和同类语言相比，真正不同的地方在哪里

## 先读什么

开始任何输出前，必须先做两件事：

1. 扫描工作区材料
2. 读取 `references/output-contract.md`

`references/output-contract.md` 定义了固定文件树、固定二级标题、最小条目数、Mermaid 规则、README 规则和完成前必须通过的校验命令。

## 固定输出合同

研究输出目录必须**严格**包含以下文件：

- `00-scope-and-plan.md`
- `01-landscape.md`
- `02-architecture.md`
- `03-representations.md`
- `04-pipeline-stages.md`
- `05-evaluation-rules.md`
- `06-runtime-mechanics.md`
- `07-end-to-end-trace.md`
- `08-failure-and-edge-cases.md`
- `09-synthesis.md`
- `README.md`

规则：

- 不允许用其他文件名替代这些文件。
- 不允许把 `03-representations.md` 和 `04-pipeline-stages.md` 折叠进 `05-evaluation-rules.md`。
- 不允许跳过某个文件并声称"内容已在别处覆盖"。
- 除 `README.md` 之外，每个阶段文件都必须至少包含一个 Mermaid 图。

## 每个文件必须做什么

- `00-scope-and-plan.md`
  - 固定研究范围、主题分类、核心问题、材料清单和输出计划
- `01-landscape.md`
  - 建立全局地图，明确语言规范结构、实现代码入口和初始问题
- `02-architecture.md`
  - 解释核心抽象、编译/运行时组件关系、数据流、分层和边界
- `03-representations.md`
  - 显式分析关键表示：AST / IR / 运行时对象 / 环境结构 / 类型表示，不得只在管线阶段中零散提到
- `04-pipeline-stages.md`
  - 显式分析管线阶段：词法分析、语法分析、类型检查、优化、代码生成等，不得只在求值规则中顺带提到
- `05-evaluation-rules.md`
  - 分析关键求值规则、类型推导规则、约束求解，以及规则之间如何协作
- `06-runtime-mechanics.md`
  - 把运行时机制单独讲清楚：GC、调度、栈管理、FFI、异常处理
- `07-end-to-end-trace.md`
  - 选代表性程序片段，追踪从源码到执行结果的完整过程
- `08-failure-and-edge-cases.md`
  - 单独分析编译错误、运行时异常、类型系统边界和诊断机制
- `09-synthesis.md`
  - 提炼设计哲学、trade-off、模式和核心结论
- `README.md`
  - 提供所有文件的相对链接、阅读顺序和一句话摘要

## 工作流

### 1. 建立研究目录

创建 `study/<topic>/`，并按固定输出合同规划文件。

### 2. 先做结构化研究，再写文档

不要一边看一边把内容随意散落到各文件里。先明确：

- 这个语言机制的核心表示有哪些
- 管线中有哪些关键阶段
- 关键求值/类型规则有哪些
- 最值得追踪的执行路径和错误诊断是什么

然后把它们分别放进对应文件。

### 3. 显式填充 `03` 和 `04`

这是本 skill 的硬要求。

- `03-representations.md` 必须显式列出并分析关键的 AST 节点、IR 表示、运行时对象或类型结构
- `04-pipeline-stages.md` 必须显式列出并分析每个管线阶段的输入、输出和职责

不要把这两类内容吞进 `05-evaluation-rules.md`。

### 4. 表示、管线、求值规则和边界条件分开写

以下四类内容必须分开：

- 表示（03）
- 管线阶段（04）
- 求值规则（05）
- 端到端 trace（07）

`08-failure-and-edge-cases.md` 也必须单独存在，不能把错误诊断和边界案例散落在其他章节里就算完成。

### 5. 完成前强制校验

在宣称研究完成前，必须运行：

```bash
python3 deep-study/scripts/validate-study.py study/<topic>
```

如果校验失败：

- 不得宣称完成
- 不得说"主要内容已经有了"
- 必须修正文档并重新运行校验，直到退出码为 `0`

## 优先收集的证据

- 语言规范、参考手册、设计文档
- 编译器或运行时源码中的关键阶段和关键数据结构
- 测试套件，尤其是 edge case 和错误诊断测试
- 小程序验证结果，如果环境允许
- 错误信息生成逻辑，因为它常常暴露规则判断点

如果"规范描述"和"实现行为"不一致，要明确区分"规定如此"和"当前实现如此"。

## 深挖时优先问的问题

- 这个规则在哪个阶段真正生效
- 一个具体程序片段是如何一步步被解释、编译或拒绝的
- 这个语言承诺为什么能成立，靠什么机制保证
- 哪个反例最能说明直觉模型的边界
- 如果换一种实现策略，会得到什么收益和代价

## 引用规则

语言机制分析根据材料类型混合使用引用方式：

- 实现源码：`path:line`
- 语言规范、参考手册：优先使用"文档名 + section / 章节号"
- 设计文档：文档标题 + 小节标题

优先使用稳定锚点，不要把所有材料机械压成同一种引用格式。

## 不合格示例

以下情况都视为不合格：

- 只写求值规则文档，没有 `03-representations.md`
- 只在执行路径里顺带提到管线阶段，没有 `04-pipeline-stages.md`
- 只列 AST 节点名称，没有写节点字段含义、不变量和阶段归属
- 只给管线流程图，没有解释每个阶段的信息变化
- 只讲正确编译路径，不讲类型错误、诊断和边界案例
- README 没有完整链接所有文件
- 没跑 validator 就宣称研究完成

## 常见失真

- 只做教材式概念总结，没有追到实现和边界案例
- 只列编译阶段名称，没有解释信息在阶段间如何转移
- 只讲规则，不讲错误、诊断和失败路径
- 用比喻替代机制本身，没有指出比喻在哪些边界会失效
- 把语言规范、实现细节和用户心智模型混在一起，不加区分

## 完成标准

只有同时满足以下条件，分析才算完成：

1. 固定文件树完整存在
2. 固定二级标题完整存在
3. 最小条目数达到要求
4. 每个阶段文件都有 Mermaid 图
5. 引用符合材料类型
6. `python3 deep-study/scripts/validate-study.py study/<topic>` 退出码为 `0`

如果以上任一条件不满足，禁止宣称完成。
